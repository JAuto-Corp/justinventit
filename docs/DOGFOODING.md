# Dogfooding Guide

> How to develop justinventit using justinventit.

## The Paradox

justinventit is a framework for AI-native development. It should be built using AI-native development. But the framework doesn't exist yet — so we bootstrap.

## Bootstrap Strategy

### Phase 1: Manual ATDD (M0-M1)
The framework's own CLAUDE.md, skills, and hooks are hand-written (not generated by Copier). We follow the ATDD cycle manually:
- Write SPEC for each roadmap item
- Write acceptance criteria (informal — no TDS yet)
- Implement → type-check (where applicable) → validate
- Emit `[FRICTION:*]` signals when the process feels wrong — these become improvements

### Phase 2: Self-Scaffold (M1)
Once M1 is complete, we can `copier copy` justinventit into a test project and run a real ATDD cycle using the generated scaffold. This is the first real dogfood test.

### Phase 3: Self-Host (M2+)
The framework's own development uses justinventit's generated scaffold. Changes to the framework are tested by regenerating the scaffold and verifying it still works.

## Development Workflow

### Starting a Session

1. Read `CLAUDE.md` — it's the routing table for this repo
2. Read `docs/ROADMAP.md` — find the next unchecked item in the current milestone
3. Read `context/WORKING.md` — check the last observation block for context

### Working on a Roadmap Item

**For template files** (things that get copied into projects):

1. Identify the file to create/modify in `template/`
2. If it's Jinja2-templated: test with `copier copy . /tmp/test --defaults`
3. If it's a hook script: test with the hook test harness (once it exists)
4. If it's a skill: verify frontmatter has `name` and `description`
5. Commit with descriptive message referencing the roadmap item

**For framework docs** (things that stay in this repo):

1. Write/update in `docs/`
2. Keep docs proportional to complexity — don't over-document simple things
3. If a doc describes a pattern that's also in a skill, the skill is authoritative

**For Copier config** (`copier.yml`):

1. Add/modify questions
2. Test all relevant combinations: `copier copy . /tmp/test -d stack=X -d database=Y`
3. Verify generated output is valid (CLAUDE.md renders correctly, hooks are executable)

### Validation

Before considering a roadmap item done:
- [ ] Template renders correctly (`copier copy . /tmp/test --defaults`)
- [ ] Hook scripts are executable and have correct shebang
- [ ] Skills have valid SKILL.md with name + description frontmatter
- [ ] No hardcoded project-specific values in template files
- [ ] Jinja2 conditionals tested with both true/false paths
- [ ] Commit message references the roadmap item

### Friction Tracking

This is where dogfooding produces the most value. As you develop justinventit:

1. **Notice friction** — "this is harder than it should be" or "I keep doing this wrong"
2. **Emit signal** — `[FRICTION:SKILL]`, `[FRICTION:HOOK]`, `[FRICTION:WORKFLOW]`, `[FRICTION:CONTEXT]`
3. **Log it** — the stop hook appends to `context/FRICTION_LOG.md`
4. **Fix it** — update the relevant skill, hook, or doc

Every friction signal during development is a direct improvement to the framework. This is the self-improvement loop in action.

### Testing Strategy

**Level 1 (manual, now):**
```bash
# Test default generation
copier copy . /tmp/test-default --defaults
# Verify output structure
find /tmp/test-default -type f | sort

# Test specific stack
copier copy . /tmp/test-nextjs -d project_name="Test" -d stack=nextjs -d database=supabase -d testing=playwright
# Verify CLAUDE.md renders
cat /tmp/test-nextjs/CLAUDE.md

# Test hook execution
bash /tmp/test-default/.claude/hooks/stop/runner.sh
```

**Level 2 (automated, M1):**
Hook test harness with mock transcripts. Copier generation matrix test.

**Level 3 (integration, M1):**
Generate scaffold for customer-portal. Diff against actual. Run ATDD cycle.

## Key Principles

1. **Eat your own cooking** — every development session follows the workflow we're building
2. **Friction is signal** — when something is hard, that's data, not noise
3. **Template-first** — changes to `template/` are the primary output; docs support understanding
4. **Test the output, not the template** — validate what Copier generates, not the Jinja2 source
5. **One milestone at a time** — don't jump to M2 features while M1 items are unchecked

## Customer-Portal as Reference Implementation

The customer-portal repo is the existence proof. When in doubt:

1. Check how customer-portal solves the problem
2. Extract the pattern (separate mechanism from content)
3. Template it (Jinja2 for stack-specific, plain for universal)
4. Test the generated output against the original

Key files to reference:
- `customer-portal/.claude/settings.json` — hook registrations (10 hooks)
- `customer-portal/.claude/hooks/stop.sh` — the 600-line stop hook (decompose into pipeline)
- `customer-portal/.claude/skills/` — 12+ skills (extract orchestrator patterns)
- `customer-portal/context/WORKING.md` — state file format (convert to append-only)
- `customer-portal/test-data/` — TDS architecture (extract story/scenario patterns)
